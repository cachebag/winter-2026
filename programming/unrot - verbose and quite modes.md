This walks you through adding `-v` / `--verbose` and `-q` / `--quiet` flags to your CLI. I'll explain what to do and why, give you the shape of each piece, and leave the actual typing to you. When you get stuck, the hints are there. When you're not stuck, skip them.

## The Big Picture

Right now, every `println!` and `io.write_str(...)` in your code runs unconditionally. You're going to introduce a concept of **verbosity level** that flows through the program and gates what gets printed. The three levels are:

| Level | Flag | Behavior |
|---|---|---|
| Quiet | `-q` | Bare minimum. In `list` mode: just paths. In `fix` mode: suppress "no candidates" noise, show only links + prompts + final summary. |
| Normal | (default) | What you have today. No change. |
| Verbose | `-v` | Extra detail. Scan progress ("scanning... found 14 symlinks so far"), candidate search stats ("searched 2301 files, found 3 matches"), filesystem op details ("removing symlink at /foo/bar"). |

The conflict rule: if someone passes both `-v` and `-q`, quiet wins. (They asked for less noise; respect that.)

**Why an enum instead of a number?** Because you'll pattern match on it. `match verbosity { Quiet => ..., Normal => ..., Verbose => ... }` reads like English. An integer (0, 1, 2) would compile, but you'd spend the rest of your life wondering whether 0 is quiet or verbose.

## Step 5: Actually use verbosity in the resolver

Now the fun part. You have the verbosity value inside the resolver. Decide what changes at each level:

### Quiet mode

In `run_immediate`, the `present(...)` call writes the header, candidates list, and action hints for every broken link. In quiet mode, you still need to show the link (the user needs to know what they're deciding about) and the prompt, but you can suppress:
- The "no candidates found" line (if there are no candidates, quiet mode could just show the link and the actions)
- The score/shared-dirs details on each candidate — show just the path

The simplest approach: write a second presentation function, or pass `verbosity` to `present()` and let it decide. I'd recommend passing it to `present()`.

**Where:** `core/src/resolver/display.rs`

**What to change in `present`:**

```rust
pub fn present(w: &mut impl fmt::Write, case: &RepairCase, verbosity: Verbosity) -> fmt::Result {
    format_header(w, case)?;
    match verbosity {
        Verbosity::Quiet => format_candidates_quiet(w, case)?,
        _ => format_candidates(w, case)?,
    }
    format_actions(w, case)
}
```

Then write `format_candidates_quiet` — just the paths, no scores, no basename counts:

```rust
fn format_candidates_quiet(w: &mut impl fmt::Write, case: &RepairCase) -> fmt::Result {
    for (i, candidate) in case.candidates.iter().enumerate() {
        writeln!(w, "  [{}] {}", i + 1, candidate.path.display())?;
    }
    Ok(())
}
```

### Verbose mode

In verbose mode, after executing a filesystem operation, print extra detail. In `format_outcome` in `session.rs`, you could add lines like:

- After relinking: `"  (removed old link, created new symlink)"`
- After removing: `"  (symlink entry deleted from filesystem)"`

Small stuff. The idea is that someone debugging a weird situation can see exactly what happened. Pass `verbosity` into `format_outcome` and add a conditional branch.

Change nothing. This is your current behavior.

## Common mistakes to watch for

- **Forgetting to update all three subcommand structs.** You have `ScanArgs`, `FixArgs`, `ListArgs` and the top-level `Cli`. All four need the flags. It's tedious. Do it anyway.

- **Making the enum `pub` but forgetting to re-export it.** If `Verbosity` is defined in `lib.rs` it's already at crate root. If you put it in its own module, add `pub use` in `lib.rs`.

- **Not passing `Verbosity` through `run_batch`.** Both `run_immediate` and `run_batch` need it. Easy to forget the batch path since you test the immediate path more often.

- **Testing with `dry_run = false` and reusing temp dirs.** Once a symlink is relinked, it's not broken anymore. Use separate temp dirs per test run, or use `dry_run = true` when you only care about output, not filesystem effects.
